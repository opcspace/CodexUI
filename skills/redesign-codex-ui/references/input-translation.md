# Input Translation

## Reference images

Inspect the whole image before focusing on isolated styling. Record only observable evidence, then separate interpretation from fact.

Use this order:

1. Map the page regions and their proportions.
2. Identify the visual hierarchy and reading path.
3. Estimate density, spacing rhythm, type roles, and contrast.
4. Inventory repeated components and interaction states that are visible.
5. Extract a restrained palette and shape language.
6. Mark details that cannot be inferred from a static image.

Do not blindly reproduce fixed pixel coordinates. Translate the composition into responsive constraints. If the image shows only one viewport, infer narrow behavior from content priority and existing product behavior.

Use a cue matrix:

| Cue | Observation | Translation |
| --- | --- | --- |
| Layout | Narrow navigation beside a broad reading pane | Fixed or collapsible rail plus fluid content |
| Density | Small type with generous group spacing | Compact controls, clear sectional rhythm |
| Color | Mostly neutral with one vivid accent | Semantic neutral scale plus one action color |
| Shape | Low-radius controls and thin separators | Reduce pill usage and elevation |

Do not copy trademarks, proprietary icons, distinctive illustrations, or brand assets unless the user owns or supplies them for reuse.

## Personas and roles

Translate a role into product decisions rather than decorative stereotypes.

Derive:

- primary jobs and repeated actions;
- information needed at a glance;
- tolerance for density and complexity;
- likely environment, device, and session length;
- terminology and default views;
- accessibility or attention constraints;
- trust signals and risk-sensitive actions.

Examples:

| Persona | Likely need | UI consequence |
| --- | --- | --- |
| Engineer | Diff clarity and fast command access | Dense navigation, strong code contrast, keyboard-first controls |
| Founder | Cross-task status and rapid prioritization | Clear task states, concise summaries, visible next actions |
| Researcher | Long-form reading and provenance | Wider reading rhythm, stable anchors, source and tool trace clarity |
| First-time user | Confidence and recoverability | Progressive disclosure, explicit labels, forgiving empty states |

Avoid gender, age, or profession stereotypes. Tie each decision to a stated or defensible workflow need.

## Named styles

Convert a style label into a finite rule set before coding:

- three to five adjectives;
- palette behavior;
- typography behavior;
- geometry and elevation;
- density and whitespace;
- icon and illustration treatment;
- motion character;
- one list of forbidden motifs.

For a named designer, brand, or product, describe high-level characteristics and create an original system. Do not claim endorsement or exact replication.

## Handling incomplete inputs

Proceed with clearly stated assumptions when the missing choice is reversible. Ask one focused question only when the answer materially changes scope, architecture, or a non-reversible outcome. A missing exact color value is usually inferable; whether to redesign only the composer or the entire app is not.
